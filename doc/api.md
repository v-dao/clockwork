# Clockwork — 接口与数据契约

本文档描述模块之间的**概念级 API 与数据边界**，用于指导实现与联调。命名与类型在编码阶段可映射为 C++20 接口；未实现前以**契约含义**为准。

---

## 1. 文档范围

| 接口域 | 说明 |
|--------|------|
| 仿真引擎 — 控制面 | 运行程序 → 引擎的指令 |
| 仿真引擎 — 态势面 | 引擎 → 态势显示 / 记录器的输出 |
| 想定 — 初始化 | 想定编辑 → 引擎的「限定信息」 |
| 仿真模型 — 能力模型 | 六类模型的职责与典型输入输出 |
| 脚本宿主 | Lua / 可视化蓝图可调用的能力边界 |
| 分布式仿真 | 多引擎跨主机数据通信，模型与分布式模型数据交互 |
| 快照 — 保存 / 恢复 | **仅本地单机**；与分布式联邦**互斥**，见第 2 节 |

---

## 2. 仿真引擎控制 API（概念）

运行程序或外部模块向引擎发送的**控制指令**（与 README 一致）：

| 指令 | 预期语义 |
|------|----------|
| **初始化** | 消费想定提供的限定信息，构建实体与模型实例；未开始时间推进。 |
| **开始** | 开始（或恢复）按仿真时钟推进。 |
| **暂停** | 冻结仿真时钟；保留运行时状态。 |
| **结束** | 停止推进并进入可释放资源状态（具体是否保留态势由实现约定）。 |
| **设置仿真倍速** | 墙钟时间与仿真时间的比例关系。 |

**建议**：用显式状态机表达引擎生命周期，避免「初始化」与「开始」语义重叠导致重复构建。

### 2.1 运行模式与快照（概念）

README 要求支持**保存快照并恢复快照**，且与产品规则一致：**仅本地单机**可用；**分布式版本 / 联邦模式下不可用**。

| 运行模式 | 快照 save / restore |
|----------|----------------------|
| **本地单机**（未加入分布式联邦、无活跃对等会话） | 允许（实现可分阶段：先内存闭环，再落盘格式）。 |
| **分布式联邦**（已与远端引擎建立数据通信或会话仍视为活跃） | **禁止**：相关 API 须**失败返回**（推荐显式错误码）；不得将单机快照恢复到已联网节点等未定义行为。 |

引擎须能可靠判定「是否处于联邦模式」（例如连接建立至会话结束期间均视为分布式）。从单机切换到联邦前，实现可要求先**结束**仿真或清理状态，避免误用残留快照状态。

### 2.2 错误码（实现：`cw::Error`）

公开 API 返回 `cw::Error`；`cw::ok(e)` 判断是否成功。可搭配 **`cw::error_code_str(e)`**（稳定英文标识，便于日志检索）与 **`cw::error_message(e)`**（简短中文说明）。引擎在失败返回前会调用 **`cw::log_error(context, e)`** 写一行诊断日志（含操作上下文与当前状态名，视调用点而定）。

| 值 | `error_code_str` | 典型场景 |
|----|------------------|----------|
| `Ok` | `Ok` | 成功 |
| `InvalidArgument` | `InvalidArgument` | 数值参数非法（如倍速 ≤ 0） |
| `IOError` | `IOError` | 读想定文件失败等 |
| `Internal` | `Internal` | 预留 |
| `NoSnapshot` | `NoSnapshot` | 未 `save_snapshot` 即 `restore_snapshot` |
| `ParseError` | `ParseError` | 想定文本解析或结构校验失败 |
| `WrongState` | `WrongState` | 当前引擎状态不允许该操作（如未 `initialize` 即 `start`） |
| `NotAllowedWhenFederated` | `NotAllowedWhenFederated` | 联邦模式下调用快照 API |
| `UnsupportedScenarioVersion` | `UnsupportedScenarioVersion` | 想定 `version` 非引擎支持的值 |

## 3. 想定初始化数据（「限定信息」）

引擎初始化阶段需要的最小信息边界（逻辑字段，非最终序列化格式）：

| 类别 | 内容示例 |
|------|----------|
| **实体** | 唯一标识、显示名称、初始位姿、所属方（若需要） |
| **平台与型号** | 引用的型号数据、2D 图标、3D 模型路径或句柄、属性表 |
| **挂载模型** | 每实体上的 `mover` / `sensor` / `comdevice` / `processor` / `weapon` / `signature` 列表及参数 |
| **静态图层** | 航线、空域几何与属性 |
| **通信网** | 节点、链路、约束（带宽、延迟等，视仿真深度而定） |
| **控制逻辑** | Lua 模块或蓝图资源引用及入口 |

序列化格式（二进制、JSON 子集、自定义文本等）由实现选定，但应满足**无第三方库**约束下的自研解析。

## 4. 仿真态势输出 API（概念）

引擎向态势显示（及可选记录、外部系统）输出的**态势快照或增量**，建议包含：

| 数据 | 用途 |
|------|------|
| **仿真时间** | 当前仿真时刻、倍速、运行状态 |
| **实体列表** | 标识、名称、位姿、姿态变化率（用于转动矢量） |
| **运动矢量** | 速度等，用于矢量箭头 |
| **几何辅助** | 实体局部坐标轴显示用 |
| **传感器 / 武器可视化** | 雷达波束、干扰波束、雷达包络等几何与样式参数 |
| **探测与交互摘要** | 可选：用于 HUD 或标签（实现阶段权衡性能） |

**原则**：显示层只读态势结构，不回调模型内部；需要交互时通过运行程序转发为**控制指令**或**想定级变更**（若支持运行时改参）。

## 5. 分布式仿真 API（概念）

README 要求仿真引擎具备**分布式仿真**能力：与其他计算机上的仿真引擎**数据通信**，使模型可与**分布式的模型数据**交互。实现阶段需约定（但不限于）：

| 主题 | 契约要点 |
|------|----------|
| **连接与成员** | 引擎实例如何发现对端、加入/退出联邦、会话标识 |
| **时间** | 仿真时钟是否全局一致、步进对齐、暂停/倍速在联邦内的传播 |
| **交换数据** | 哪些实体状态 / 量测 / 通信内容可跨节点订阅或发布；与本地 `sensor`、`comdevice`、`signature` 等模型的衔接点 |
| **可靠性** | 丢包、延迟对仿真的容忍策略；是否需要确定性重放 |
| **安全** | 传输与鉴权（若跨不可信网络） |
| **快照** | **不提供**：联邦激活期间禁止保存/恢复快照（与第 2.1 节一致）。 |

分布式链路与**无第三方依赖库**约束下，协议与序列化需自研或在政策允许范围内最小实现。

## 6. 仿真模型类型契约

六类模型与引擎的协作方式（逻辑层面）如下。

| 类型 | 引擎提供给模型 | 模型回报引擎 / 其他模块 |
|------|----------------|-------------------------|
| **mover** | 时间步长、环境约束、控制输入 | 新位姿、速度、加速度等 |
| **sensor** | 世界实体列表 / 空间索引查询接口、signature 数据 | 探测列表、量测 |
| **comdevice** | 网络拓扑、入站报文 | 出站报文、链路状态 |
| **processor** | 传感器/通信等汇总输入 | 内部状态、对 mover/weapon 等的决策输出 |
| **weapon** | 交战规则参数、目标指示 | 发射事件、弹药实体创建请求 |
| **signature** | — | 各频段/通道特征供 sensor 与显示引用 |

具体每帧**更新顺序**应在架构或引擎设计中写死或配置，以保证确定性复现。

## 7. 脚本与蓝图宿主 API（概念）

Lua 与可视化蓝图共享**同一语义边界**时更易维护：

| 能力 | 说明 |
|------|------|
| **只读查询** | 实体自身状态、传感器结果、报文摘要等（需定义安全子集） |
| **意图下发** | 机动目标点、传感器开关、武器允许、通信发送等（由引擎校验并转给模型） |
| **错误处理** | 脚本异常不得破坏引擎进程；应记录并可选停用该实体逻辑 |

可视化蓝图到引擎的映射（节点类型、数据_pin）在实现阶段单独成表；本文件只要求**能力集合与 Lua 对齐**。

---

## 8. 态势显示输入 API（概念）

态势显示模块的输入应至少包括：

- 引擎输出的**态势流**（见第 4 节）。
- 想定中的**静态图层**（航线、空域），可在初始化时一次性载入并与引擎标识关联。

可选：用户交互事件（选中的实体、相机参数）由运行程序消化，不强制进入引擎。
