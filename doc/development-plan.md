# Clockwork — 开发计划

本文档在 [README.md](../README.md)、[requirements.md](requirements.md)、[architecture.md](architecture.md)、[api.md](api.md)、[development.md](development.md) 基础上，梳理**推荐实施顺序**、交付物与验收要点。周期为估算量级，可按人力调整。

---

## 1. 目标与硬约束

| 约束 | 对计划的影响 |
|------|----------------|
| C++20、g++ | 首阶段固定工具链与编译脚本；优先标准库 |
| `cmd/<程序名>/*.cpp`、`lib/<库名>/*.cpp` | 先划库边界（引擎核、数学/几何、渲染封装、想定序列化等），再落程序入口 |
| **不使用第三方依赖库** | 数学、网络协议、脚本（若采用 Lua）、文件格式均需自研或内置源码；分布式传输层需自研 |
| 界面类封装 + **OpenGL** | 渲染与窗口抽象单独成库，避免业务代码直接绑死平台 |
| 分布式仿真 | 在单机引擎与模型管线稳定后并行设计协议，避免早期返工 |
| **快照与恢复**（README） | **仅本地单机**：须支持保存/恢复快照；**分布式模式下禁止**快照（与联邦时间、跨节点状态不一致风险）；实现上需显式运行模式标志并在 [api.md](api.md) 中固定语义 |

## 2. 阶段总览

| 阶段 | 名称 | 核心产出 | 建议依赖 |
|------|------|----------|----------|
| 0 | 工程与公共基础 | 构建、目录、`lib` 分层、日志/断言、基础数学与类型 | — |
| 1 | 仿真内核骨架 | 引擎状态机、时钟与倍速、实体/模型容器、空模型桩、态势快照结构；**单机模式下**快照保存/恢复 API 与内存态闭环；分布式标志占位（阶段 9 接通） | 0 |
| 2 | 想定数据与初始化 | 自研想定交换格式（或最小可读格式）、加载管线、限定信息 → 实体实例化 | 0–1 |
| 3 | 模型 MVP | `mover`、`signature` 最简实现；`sensor` 规则探测；其余类型桩或极简版 | 1–2 |
| 4 | 态势显示 MVP | OpenGL 窗口封装、2D 或 3D 先通一路、订阅引擎态势、实体/航线/空域 | 1–2 |
| 5 | 仿真运行程序 | 组合引擎 + 显示、控制指令 UI、基础相机与选实体 | 3–4 |
| 6 | 模型扩展 | `comdevice`、`processor`、`weapon` 与仿真步内调度顺序固化 | 3–5 |
| 7 | 脚本与蓝图 | Lua 宿主（若立项）或蓝图 VM/编译最小闭环；与引擎意图 API 对齐 | 1、6 |
| 8 | 想定编辑（可用版） | 型号/平台/部署/航线/空域/通信网/逻辑绑定；导出与运行程序联调 | 2、5 |
| 9 | 分布式仿真 | 对等连接、时间对齐策略、交换数据契约实现；与 `sensor`/`comdevice` 衔接；**运行时关闭并拒绝快照 API** | 1、6 |
| 10 | 深化与质量 | 雷达/干扰/包络可视化、性能、确定性复现、**单机快照**文件格式与版本迁移、文档与示例想定 | 全程 |

阶段 3与 4 可在人力允许时**并行**（同一引擎接口版本下联调）。

## 3. 分阶段任务说明

### 阶段 0：工程与公共基础

- 建立 g++ + **Ninja** 编译规则（目标文件与可执行程序、C++20、警告级别；根目录 `build.ninja`）。
- 约定命名空间、错误码、日志策略；可选最小单元测试框架（自研或用不违反「第三方库」政策的方案需集体决策）。
- `lib` 中拆分建议：`math`（向量、四元数、地理/投影若需要）、`core`（基础类型）、后续 `engine`、`render`、`scenario` 等按依赖树增长。

**验收**：空 `main` 可编译链接；公共库可被多个 `cmd` 链接。

### 阶段 1：仿真内核骨架

- 实现引擎控制指令：**初始化、开始、暂停、结束、设置仿真倍速**（状态机清晰）。
- 仿真时钟：固定步长或可变步长策略（先固定步长降低复杂度）。
- 实体 ID、模型挂载表、每 tick 调度：`pre tick → 各类模型 → 聚合态势`。
- 定义**态势快照**内存结构（与 [api.md](api.md) 第 4 节一致方向），先填占位字段。
- **保存快照 / 恢复快照**（README，**仅单机模式**）：阶段 1 至少完成**进程内**闭环——在**非分布式**运行配置下，捕获可复现运行态（仿真时间、倍速、实体与挂载模型及与运动相关的占位状态等），`restore` 后在不加载想定文件的情况下可继续 `step`并与保存前一致；分布式标志为真时，相关 API 返回错误。**落盘格式、版本号、与想定引用的对齐**在阶段 2/10 收紧（见里程碑 M1）。

**验收**：无 UI 的 `cmd` 驱动引擎多步推进，日志打印实体数量与仿真时间；**同一 `cmd` 或测试**在单机模式下演示保存快照 → 恢复 → 继续推进，仿真时间与实体计数一致（分布式验收见阶段 9，不包含快照）。

### 阶段 2：想定数据与初始化

- 定义想定文件（或流）字段：**实体、型号引用、模型参数、位姿、航线/空域/通信网、脚本引用**（见 [api.md](api.md) 第 3 节）。
- 解析器自研；版本号字段预留。
- 「初始化」指令路径：加载 → 校验 → 构建运行时对象。

**验收**：一份手写最小想定可启动并生成至少一个带 `mover` 的实体。

### 阶段 3：模型 MVP

- **mover**：常速或简单积分更新位姿；可选航线点插值（依赖想定航线数据结构）。
- **signature**：最简电磁/光学截面或标签字段，供 `sensor` 使用。
- **sensor**：距离/视场规则 + signature 输入，输出探测列表；结果进入态势。
- 其余类型：空实现或返回默认值，接口齐备。

**验收**：多实体运动 + 探测结果在态势结构中可观察（日志或调试导出）。

### 阶段 4：态势显示 MVP

- 窗口与上下文、OpenGL 初始化封装（`lib` 界面/渲染层）。
- 先实现**一种**视图：2D 地图投影或简化 3D，避免双轨并行拖期。
- 绘制实体位置、速度矢量（数据来自态势）；航线/空域来自己想定静态层或引擎下发。

**验收**：运行程序或独立 viewer 以固定帧率拉取/接收态势并绘制。

### 阶段 5：仿真运行程序

- `cmd` 入口：嵌入引擎与显示循环；线程模型选定（单线程先跑通）。
- UI：开始/暂停/倍速/结束；可选简单实体列表。
- 人机交互与引擎指令映射（见 [api.md](api.md) 控制面）。

**验收**：端到端：加载想定 → 运行 → 屏幕可见运动；可暂停恢复。

### 阶段 6：模型扩展

- **comdevice**：报文队列与拓扑；与想定通信网一致。
- **processor**：汇总传感器/通信，写状态供脚本或 weapon使用。
- **weapon**：发射事件、创建新实体（弹药）流程与引擎对象池。
- 文档化**模型更新顺序**（与 [architecture.md](architecture.md) 一致）。

**验收**：最小交战/通信场景可复现；同一步内行为确定。

### 阶段 7：脚本与蓝图

- **Lua**：若允许链接/嵌入，实现沙箱边界（[api.md](api.md) 第 7 节）；与「无第三方库」冲突则改为 Lua 源码内置方案或纯自研 DSL。（**当前日历暂缓**，见 §5「Lua 脚本（暂缓）」。）
- **可视化蓝图**：存储格式、运行时解释或编译为指令序列；与 Lua **能力对齐**。（**不暂缓**，按本阶段与里程碑推进实装。）

**验收**：同一行为可用脚本驱动 mover/sensor 开关；异常不崩溃引擎。

### 阶段 8：想定编辑（可用版）

- 型号参数表单、平台组装、场景部署、航线/空域/通信网编辑、逻辑绑定。
- 导出与阶段 2 格式一致；与运行程序闭环。

**验收**：非程序员可完成一份想定并由运行程序加载（培训级演示）。

### 阶段 9：分布式仿真

- 传输层：在约束下自研（如 TCP 会话 + 长度前缀帧）；对等发现可简化为配置文件。
- **时间**：主从或步进屏障；暂停/倍速传播。
- **数据**：实体状态/量测/通信子集的订阅发布；与本地模型接口（[api.md](api.md) 第 5 节）。
- **快照**：联邦激活期间**禁用**保存/恢复快照；若从单机切换为分布式，须明确是否要求先 `end` / 清状态，避免误用单机快照。

**验收**：两台机器各一引擎，同一想定子集，探测或通信可跨节点看见一致结果；**快照相关 API 在分布式模式下均不可用**（调用失败或可文档化的 no-op 策略二选一，推荐显式失败）。

### 阶段 10：深化与质量

- 雷达/干扰波束、包络几何与着色；3D 地球/2D 地图双模式若未做则补齐。
- 性能剖析：实体规模、带宽；**仅单机**快照与「记录/回放」联动（文件级与大数据量压测不覆盖分布式）。
- **快照文件**（单机）：校验和、版本迁移策略、与引擎/想定版本的兼容性说明。
- 示例想定、与 README 场景类型对齐的演示数据。

## 5. 风险与待定决策

- **Lua/蓝图是否违反「无第三方」**：需在阶段 0 前书面定案；影响阶段 7 排期。
- **2D 与 3D 谁先**：建议先单一路径打通，再扩展，避免渲染双栈拖垮阶段 4–5。
- **分布式与单机 API**：尽早冻结态势与交换 DTO 版本，减少阶段 9 破坏性改动。
- **快照语义（仅单机）**：保存时是否允许 `Running`、恢复后默认 `Paused` 还是 `Ready` 等，需在实现前写入 [api.md](api.md)；**与分布式互斥**已定为产品规则，联邦模式下不得使用快照。
- **Lua 脚本（暂缓，2026-04）**：当前排期**不实现** Lua 宿主、不执行想定中 `entity_script`（`kind` 为 `lua` / `lua_script`）的逻辑；想定格式可继续保留该字段占位，待后续再实装与联调。
- **可视化蓝图（不暂缓）**：**不与 Lua 同步推迟**；仍按上文**阶段 7**与里程碑 **M3（阶段 6+7）**推进：存储格式、运行时解释或编译管线、与引擎意图 API 对齐；想定中 `entity_script`（`blueprint` / `bp`）为正式排期能力，而非占位暂缓项。
